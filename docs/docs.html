W tym rozdziale skupimy się na instalacji i konfiguracji systemu. Co prawda system posiada gotowy instalator, który powinien poprowadzić krok po kroku w procesie instalacji systemu, lecz skupimy się na problemach, które można napotkać w trakcie konfiguracji.

Wymagania sprzętowe

Jak każda aplikacja PHP, system do działania potrzebuje interpretera PHP, w wersji 5 (zalecana wersja 5.2). Przydatny może okazać się również serwer ;) Polecamy popularny serwer Apache 2.0.X. Jeżeli Twoja aplikacja w domyśle ma korzystać z bazy danych, musisz zainstalować oraz skonfigurować odpowiedni system bazodanowy.

Important

Obecnie serwis 4programmers.net działa pod bazą danych MySQL 5.0.

Bardzo zalecane jest również, aby serwer HTTP obsługiwał pliki .htaccess oraz moduł mod_rewrite.

Dodatkowo do pełnej kompatybilności z systemem, zalecane jest posiadanie następujących bibliotek PHP:

  • JSON
  • mbstring
  • XSL
  • SimpleXML

Note

W wersji MySQL 5.0.2 jedynie użytkownicy z prawami SUPER PRIVILEGE mogą tworzyć triggery, które są wykorzystywane w projekcie.

Note

Jeżeli posiadasz serwer IIS, firmy Microsoft, możesz wykorzystać moduł ISAPI_Rewrite, który obsługuje mod_rewrite na tego typu serwerach.

Najnowsze kody źródłowe

Najnowsze kody źródłowe, zarówno frameworka jak i serwisu 4programmers.net, można znaleźć na serwerze SVN: 

svn://4programmers.net/coyote/trunk

Możliwy jest również dostęp do kodów źródłowych via WWW poprzez adres: http://4programmers.net/trac/coyote. Wszelkie aktualizacje dokonywane na serwerze SVN, automatycznie dokonywane są na naszym serwerze deweloperskim - http://dev.4programmers.net. Tam również możesz testować najnowszy kod źródłowy.

Instalator

Po ściągnięciu kodów źródłowych z naszego serwera SVN, należy je rozpakować i umieścić w katalogu serwera. Na moim lokalnym serwerze, system wskazuje URL http://127.0.0.1/4programmers.

Po wpisaniu tego adresu w przeglądarce, powinien uruchomić się instalator projektu, który poprowadzi przez wszystkie etapy instalacji. Pierwszym krokiem będzie sprawdzenie konfiguracji. Zalecane jest, aby wszystkie pola były zaznaczone na zielono. Zapewnia to możliwie największą kompatybilność z systemem 4programmers.net.

W kolejnych krokach instalator poprosi o wprowadzenie informacji dotyczących bazy danych (login, hasło itp) oraz nazwy admnistratora. Możemy podać dowolne dane - system utworzy wówczas właściwe konto w bazie danych, które od tej pory będzie kontem administratora. Posiada on wszelkie możliwe uprawnienia do zarządzania systemem.

Warning

W trakcie działania instalatora, zapisuje on konfiguracje do plików XML, znajdujących się w katalogu /config. Jeżeli pracujesz na systemach *nix, upewnij się, iż katalog posiada odpowiednie prawa do zapisu.

Problemy z importem struktury bazy danych

Możliwe jest wystąpienie błędów związanych z importem struktury tabel baz danych przez instalator. Takie problemy prosimy zgłaszać na naszym tracu. Koniecznie podaj konfiguracje Twojego systemu - tj. wersja bazy danych oraz PHP.

W ostateczności konieczny będzie ręczny import struktury do bazy danych. Można to zrobić korzystając z phpMyAdmina' lub bezpośrednio z konsoli: 

mysql -u UZYTKOWNIK -p BAZA_DANYCH < /sciezka/do/pliku/sql

Note

Struktura tabel bazy danych znajduje się w pliku /module/install/schema.sql natomiast podstawowe dane znajdują się w pliku /module/install/data.sql

Konfiguracja systemu

Wszelkie informacje dotyczące konfiguracji projektu znajdują się w plikach XML, które możesz znaleźć w katalogu /config.

Warning

Upewnij się, iż dostęp do tego katalogu będzie niemożliwy z zewnątrz. Domyślnie w katalogu znajduje się plik .htaccess, który blokuje dostep z zewnątrz.

Plik config.xml

Zawiera podstawowe informacje o projekcie, takie jak host na jakim działa aplikacja, wersje aplikacji oraz informacje o tym, czy system jest zainstalowany.

Domyślnie host, nie jest wpisany w konfiguracje projektu, gdyż system automatycznie rozpoznaje nazwę serwera na którym działa. Jeżeli chcesz możesz dodać informację o hoscie, modyfikując plik i dodając węzeł XML: 

<host>serwer.com</host>

Poniższa tabela zawiera informacje o domyślnych elementach pliku konfiguracyjnego.

Table 1.1. Elementy pliku konfiguracyjnego config.xml

install Wartość true oznacza, że projekt został zainstalowany.

Plik db.xml

Na serwerze SVN, w katalogu /config, znajduje się plik db.xml.default. Zawiera on domyślną konfigurację związaną z bazą danych. Instalator projektu automatycznie utworzy w tym katalogu plik db.xml, który powinien zawierać parametry łączenia z bazą danych. W tym:

  • db_server - serwer bazy danych
  • db_login - Login użytkownika bazy danych
  • db_password - hasło umożliwiające połączenie z bazą
  • db_database - nazwa bazy danych
  • db_port - (opcjonalnie) port na którym działa baza danych
  • auto_db_connect - automatyczne łączenie z bazą danych (konieczna wartość TRUE)

Plik trigger.xml

Plik zawiera deklaracje triggerów. Więcej informacji o triggerach, znajdziesz w dokumentacji projektu Coyote. Domyślnie, plik trigger.xml zawiera instrukcje nakazujące sprawdzenie, czy system jest zainstalowany. Instalator projektu w procesie instalacji, usunie tę dyrektywę. Jeżeli instalujesz system ręcznie, należy usunąc plik trigger.xml, natomiast plik trigger.xml.copy przemianować na trigger.xml.

Plik route.xml

Plik route.xml zawiera deklaracje mechanizmu routingu. Więcej informacji na temat tego mechanizmu, można znaleźć w dokumentacji frameworka Coyote. Jeżeli instalujesz projekt ręcznie, ten plik nie wymaga zmian.

Plik autoload.php

Domyślnie w katalogu /config znajduje się plik autoload.php.default. W ręcznym procesie instalacji, jego nazwa powinna zostać zmieniona na autoload.php. Plik ten zawiera informacje o składowych systemu takich jak biblioteki oraz helpery, które są automatycznie ładowane w momencie startu systemu. Zawartość tego pliku nie powinna być modyfikowana.

Kodowanie

Dobre standardy kodowania są potrzebne w każdym większym projekcie. Szczególnie jeżeli nad danym kodem pracuje kilku programistów. Jeżeli każdy z nich używałby innych standardów, nazewnictwa - powstałby bałagan w kodzie. Tego chcemy uniknąc ustalając z góry standardy jakimi będziemy się posługiwać.

Formatowanie kodu PHP

Pliki PHP

Pliki PHP posiadają rozszerzenie .php. Jeżeli to konieczne zapisywane są w formacie UTF-8. Kod PHP zawarty jest pomiędzy instrukcje <?php ?> . Wyjątkiem jest kod widoków, gdzie możliwe jest używanie skróconego zapisu <?= ?> . Jednakże należy mieć na uwadze, iż taki zapis może być wyłączony na danej konfiguracji PHP.

Wcięcia i długość linii

W róznych projektach, różnie przyjmuje się zapisywanie wcięć. W niektórych wcięcie stanowi dwie spacje - w niektórych 4 spacje. W projekcie Coyote jako wcięcie stanowi znak tabulatora. Istnieją zalety oraz wady takiego zastosowania. Przeciwnicy używania tabulatorów jako wcięć, argumentują iż na różnych edytorach tekstu, znak tabulacji jest prezentowany w całkiem inny sposób. Istnieją jednak edytory (EditPlus), które umożliwiają określenie w opcjach, jak duże wcięcie ma stanowić tabulator.

Długość pojedyńczej linii nie powinna przekraczać 80 znaków. Nie jest to jednak rygorystyczny wymóg. Po prostu, dłuższe linie utrudniają analize i czytanie kodu.

Nazewnictwo

W Coyote stosujemy angielski zapis kodu. Tzn. nie nazywamy zmiennych $id_uzytkownika tylko $user_id. Tyczy się to również nazw klas, metod czy funkcji.

Nazwy plików

Pliki powinny posiadać rozszerzenia .php. Nazwy plików piszemy małymi znakami - np. foo.php, bar.php zamiast Foo.php. Dodatkowo biblioteki frameworka (gromadzone w katalogu /lib) powinny posiadać końcówkę .class - np. foo.class.php.

Analogicznie pliki helperów - bar.helper.php (z katalogu /helper) oraz pliki językowe (z katalogu /i18n) - foo.lang.php oraz modele (z katalog /model) - foo.model.php.

Nazwy klas

Dla wyróżnienia, pierwsza litera w nazwie klasy powinna być zapisana wielką literą - np. Lang, Config itp. Jako separator w nazwie, powinno być użyte podkreślenie - np. Front_Controller, zamiast FrontController. Wyjątkami są nazwy klas wyjątków, które zapisujemy: SqlErrorException, UserErrorException (z końcówką Exception).

Nazwy interfejsów

Nazwy interfejsów powinna poprzedzać wielka litera I - np. IConfig, IDb.

Nazwy metod

Począwszy od wersji 0.9.4-rc2 zmieniono sposób zapisu nazw metod. Zamiast znaku podkreślenia (_) stosuje się styl wielbłądzi, jednak pierwszy znak w nazwie musi być pisany mała literą - np. getByName(), findUser() zamiast get_by_name() oraz find_user(). Stare nazewnictwo może jeszcze figurować w niektórych bibliotekach, lecz zostanie poprawione w kolejnych wersjach.

Dopuszczalne jest poprzedzanie nazw metod znakami __ (np. w kontrolerach) dla podkreślenia, iż dana metoda mimo, iż jest oznaczona jako public, jest metodą pomocniczą (wykorzystywaną np. jako metoda zwrotna lub metoda wykorzystywana przez żądania AJAX).

Nazwy funkcji

Cały framework jest praktycznie obiektowy. Oparty o model MVC. Z funkcjami mamy do czynienia bardzo rzadko - praktycznie jedynie w helperach. Wówczas dopuszczalne jest użycie znaku podkreślenia (_) zamiast stylu wielbłądziego - np. html_img(), information_box().

Nazewnictwo zmiennych i pól

W nazwach zmiennych oraz pól używamy małych znaków. Dopuszczalne jest użycie podkreślenia (_) jako separatora.

Kodowanie

Łańcuchy

Stosujemy apostrofy tam gdzie to jest możliwe. 

$a = 'Hello World';

W miare możliwości używamy operatora "." pozostawiając jedną spację pomiędzy wyrazami: 

$a = 'Hello ' . 'World';

Tam gdzie jest to konieczne (w celu zwiększania czytelności) możemy używać apostrofu: 

$a = 'Mam na imię ' . $name . ', lat: ' . $age . ', mam ' . $height . ' cm wzrostu';
$a = "Mam na imię $name, lat: $age, mam $height cm wzrostu"; // czytelniej

Deklarowanie klas

Klasy muszą być deklarowane zgodnie z nazewnictwem opisanym w poprzedniej części. Dodatkowo klamra otwierająca musi znajdować się w kolejnej linii: 

/** 
 * Komentarz
 */
class Foo
{
// metody
}

class Foo {
// Źle
}

Pomiędzy słowem kluczowym class, a nazwą klasy musi się znaleźć jedna spacja przerwy. Dobrym zwyczajem jest umieszczanie komentarza opisującego działanie klasy, zgodnie z konwencją akceptowaną przez oprogramowanie PHPDoc.

Pola klasy muszą być oznaczone jako private, public lub protected: 

// Dobrze
class Foo
{
    private $bar;
    private $foo;
}

// Źle
class Foo
{
    private
        $bar;
        $foo;
}

To samo tyczy się nazw metod. Należy pamiętać o klamrze otwierającej, która powinna znajdować się w kolejnej linii: 

class Foo
{
    public function Bar() // Dobrze
    {
    // instrukcje
    }
    
    function FooBar() {  // Źle
    }
}

Instrukcje warunkowe

Należy pamiętać o umieszczaniu klamry w nowej linii oraz o wcięciu w wielkości jednego tabulatora. Oto przykład: 

if ($x == 10)
{
    if ($y == 'FOO')
    {
        $z = 10; // dobrze!
    }
}
else if ($x > 20)
{

}

if ($a == 45) { 
$b = 10;  // Źle
}

Instrukcja switch: 

// Dobrze!
switch ($value)
{
    case 10:
        $x = true;
    break;

    case 20:
        $x = false;
    break;
}


switch ($value)
{
case 10:
$x = true;
break; // Źle
}
